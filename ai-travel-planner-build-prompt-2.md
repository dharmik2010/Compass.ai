# Build Prompt: AI Travel Planner ("Plan Smarter. Travel Better.")

You are an expert full-stack engineer building a production-grade AI Travel Planner web app. Follow this spec exactly. Build incrementally (auth → core planner → itinerary engine → live features → admin), and confirm each module works before moving to the next. Use structured, typed data everywhere — never let raw LLM text hit the database or UI directly.

## 0. Architecture Principles (non-negotiable)

1. **Dual-Engine System**
   - **Deterministic engine**: geospatial routing (Google Maps), live data (weather/currency APIs), math/analytics (budget charts), and initial candidate matching (MongoDB Atlas Vector Search). Fast, cheap, precise — no LLM calls for these.
   - **Generative engine (LLM)**: called only in **JSON mode** (strict `responseSchema` / structured output). Its job is to synthesize deterministic data into personalized narrative content, run the conversational assistant, and apply contextual overrides. It never free-forms structural data.
2. **Self-Healing State Machine**: any triggering event (weather alert, hotel swap, budget cut) performs an **atomic mutation** on the Trip document (e.g. `itinerary.$[day].schedule.$[slot]`) rather than regenerating the whole trip. The frontend subscribes to the updated trip state and re-renders the timeline, map, budget charts, and packing list reactively.
3. **Token efficiency**: use vector search to narrow candidates before any LLM call; cache weather/currency responses; never regenerate a full itinerary when a partial mutation will do.
4. **Reliability**: all LLM calls must use a strict JSON schema so malformed output can never crash the app. Validate every LLM response against the schema server-side before writing to the DB; retry once on validation failure, then fall back to the last-known-good state.

## 1. Tech Stack

- **Frontend**: React (Vite), Tailwind CSS, Framer Motion, Recharts
- **Backend**: Node.js + Express, JWT auth (HTTP-only cookies), WebSockets for reactive state sync
- **Data**: MongoDB Atlas (with Vector Search enabled), Cloudinary for image hosting
- **AI**: Gemini API or OpenAI API, called strictly with JSON-mode/structured outputs
- **External APIs**: Google Maps/Places, OpenWeather, ExchangeRate API, Unsplash
- **Deployment**: Frontend on Vercel; backend on Render/Railway; MongoDB Atlas cloud cluster

## 3. Build Order & Modules

### Module 1 — Landing Page
- Hero with a single destination search bar (Google Places autocomplete)
- Two primary CTAs: "Explore Destinations" (AI discovery) and "Plan My Trip" (agentic onboarding)
- Sections: curated/trending destination cards, AI feature highlights, testimonials, "How It Works" 3-step guide
- Background: subtle looping video or Framer Motion animation — keep performant

### Module 2 — Authentication
- Email/password signup+login, Google OAuth 2.0, forgot-password flow
- Two-factor OTP + email confirmation via Nodemailer
- JWT stored in HTTP-only cookies; authenticated users land on their Trip Dashboard

### Module 3 — Agentic Trip Planner (multi-step, not a giant static form)
Collect progressively, adapting later steps to earlier answers:
- **Basics**: source, destination, dates (+ flexible-dates toggle), number of days
- **Travelers**: adults / children / seniors counters
- **Budget**: Framer Motion slider ₹5,000–₹5,00,000+, plus Budget/Standard/Luxury tier filter
- **Archetype** (multi-select chips): Family, Solo, Couple, Friends, Business, Adventure, Honeymoon, Backpacking, Luxury, Road Trip, Pilgrimage, Photography, Wildlife, Camping
- **Interests** (multi-select): Food, Nature, Adventure, Beaches, Mountains, Nightlife, Museums, Shopping, History, Local Culture, Snow, Temple, Trekking, Water Sports, Wildlife, Photography, Relaxation
- **Logistics**: accommodation type, transit mode, dietary filter (Veg/Non-veg/Jain/Vegan/Halal/None), accessibility needs (wheelchair, baby-friendly, pet-friendly, senior-friendly)
- **Free text**: special requirements box (e.g. anniversary, timing preferences)
- On submit: run vector search against `destinations` (if no destination set) → pass structured payload to LLM in JSON mode → validate → write Trip document.

### Module 4 — AI Destination Discovery ("Inspire Me")
- Inputs: current location, max budget, weather preference, topography preference, passport availability, travel month
- Pipeline: vector search narrows candidate destinations → LLM (JSON mode) adds justification text only
- Output cards per destination: name, semantic justification, current weather snapshot, estimated total cost, best time to visit

### Module 5 — Dynamic Itinerary Generator
Each day must structurally decompose into:
```
Day X
 ├─ Morning   → Activity card + travel-time vector
 ├─ Afternoon → Dining spot + cost
 ├─ Evening   → Attraction card + geolocation cluster
 └─ Night     → Hotel return + budget update
```
Every activity node carries: time slot, estimated cost, distance, and a map route reference. Generate this as validated structured JSON, not prose.

### Module 6 — Interactive Timeline UI
- Vertical Day 1 → Day N timeline (Framer Motion)
- Each day card expands to reveal transit details, booking refs, activity descriptions

### Module 7 — Budget Estimator & Tracker
- Pre-trip: Recharts pie + bar breakdown across Flights, Hotels, Food, Transport, Activities, Shopping, Taxes, Emergency Fund
- In-trip: live expense entry, instant balance recalculation, progress bar, forecast-vs-actual comparison

### Module 8 — Smart Packing Assistant
- Auto-generates checklist from destination climate, trip length, traveler demographics, selected activities
- Categories: Clothes, Shoes, Essentials, Medical, Tech
- Full CRUD on items + client-side PDF export

### Module 9 — Weather Widget
- OpenWeather integration: current conditions, 7-day forecast, temperature, humidity, precipitation chance, sunrise/sunset

### Module 10 — Currency Converter
- Auto-triggers for international destinations; ExchangeRate API; shows native vs local cost

### Module 11 — Local Recommendation Engines
- **Attractions**: Unsplash images, ratings, hours, distance, ticket price
- **Food & Dining**: breakfast/lunch/dinner picks, local delicacies, filtered by dietary preference, with reviews
- **Stay & Transport**: filtered lodging tiers with pricing/amenities/photos; transit comparison (time + fare) across flight/train/bus/metro/car rental

### Module 12 — Contextual AI Assistants (Self-Healing Logic)
- **Weather Advisor**: on high rain probability, atomically swap affected outdoor activities for indoor fallbacks (`isIndoorFallback: true`) in the DB; push update over WebSocket — do not regenerate the whole trip
- **Trip Optimizer**: geospatial clustering / distance-matrix reordering of same-day stops to cut backtracking
- **AI Chatbot + Inline Regenerator**: conversational sidebar (natural language: "cut my budget 20%", "what should I wear") plus explicit action buttons (Regenerate Day, Swap Hotel, Adventure/Relaxation Mode) that trigger the same atomic mutation endpoints
- **Safety & Etiquette Guide**: emergency numbers, nearest hospital/police/embassy, cultural tips, scam warnings, dress codes
- **Photo Spot Finder**: notable photo spots, golden-hour times, local drone rules

### Module 13 — Map Integration
- Google Maps or Leaflet; polyline routes from hotel to each day's ordered attractions; live navigation updates

### Module 14 — Engagement, Export & Admin
- Notifications: upcoming trip reminders, weather disruption alerts, packing milestones, check-in reminders
- Export: WhatsApp share, QR code generation, email itinerary, Excel/PDF export
- Reviews: post-trip ratings, markdown reviews, photo upload to object storage
- Admin panel: user/trip/destination analytics, feedback logs, system health, full AI prompt/token logs

## 4. LLM Call Contract

Every LLM call must:
1. Include a system instruction restricting output to JSON matching a defined schema (no prose, no markdown fences).
2. Be validated server-side against that schema before touching the DB.
3. Be scoped narrowly (e.g. "write narrative + justification for these 3 pre-selected destinations," not "invent destinations").

## 5. Definition of Done (what to demo to judges)

- Show vector search narrowing candidates before any LLM call (token efficiency).
- Show a forced malformed-LLM-response scenario being caught by schema validation without crashing.
- Show one itinerary event change (e.g. simulate a rain alert) mutating the DB atomically and reactively updating the timeline, map, budget chart, and packing list without a full page refresh or full regeneration.
